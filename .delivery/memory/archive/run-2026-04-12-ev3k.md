## What went well

1. **Cross-artifact consistency lesson from ga4x was applied and held.** The previous pipeline identified that Design artifacts needed a cross-artifact consistency check before DoD. This time, Design passed clean on the first round -- no localStorage key mismatches, no naming inconsistencies. A lesson learned is only a lesson if it changes the next outcome. This one did.

2. **Architect skip was correct and saved a full stage cycle.** The feature is purely UI-side: new data config, new component, sidebar refactor, calendar link utilities. No new API endpoints, no infrastructure changes, no cross-module integration. Solomon's architecture was not needed and the pipeline correctly identified that. Not every wall needs a foundation redesign.

3. **Development self-correction caught real gaps, not false positives.** QA DoD round 1 found two legitimate missing test areas: empty EVENTS array behavior (E-2 AC-10) and keyboard navigation coverage (E-3/E-5). The developer added 6 targeted tests in round 2, bringing the total from 73 to 79. The QA gate worked as designed -- it did not rubber-stamp, it inspected.

4. **Data-driven SideBar refactor future-proofs event additions.** Moving from hardcoded props to an imported EVENTS array means the next church event (Wednesday Bible Study, youth group, etc.) requires only a config entry -- no component surgery. Every family shall build in front of their own house, and the next family's plot is already surveyed.

5. **RFC 5545 compliance was thorough.** VTIMEZONE with STANDARD/DAYLIGHT transitions, CRLF line endings, line folding at 75 octets, TZID on DTSTART/DTEND -- the .ics generation followed the spec rather than taking shortcuts. Calendar interop is one of those areas where "close enough" means "broken on Outlook."

## What could be improved

1. **No runtime test execution in the pipeline.** 79 tests were written but never executed. Node.js was not available in the development environment, so all tests are CODE_COMPLETE but not VERIFIED. The standing elder rule -- "An item is not done until it is tested in production" -- applies, but so does "trust but verify." A pipeline that produces untested tests is carrying risk it cannot quantify. The team needs a CI step that runs tests before UAT, not after deployment.

2. **Checkpoint removal mid-pipeline creates audit gaps.** The stakeholder removed all human gates with "keep the team working until done." The builders were trustworthy this time, but removing every checkpoint means the pipeline's quality assurance is entirely self-referential -- agents reviewing agents with no external validation until production. For a church website the risk is low; for any pipeline handling sensitive data or compliance requirements, this pattern should not be repeated without explicit acknowledgment of the tradeoff.

3. **Refine needed an eval-opt round that Idea could have prevented.** The QA evaluator found gaps in the PRD that required a revision cycle. If the Idea brief had included more explicit acceptance criteria guidance (particularly around empty states and keyboard navigation edge cases), the PRD might have covered those on the first pass. Front-loading specificity at Idea compresses Refine -- the ga4x brainstorm lesson applies here too.

## Lessons learned

1. **Cross-artifact consistency checks work when institutionalized.** The ga4x retrospective recommended pre-DoD consistency linting. This pipeline applied it and Design passed clean. The lesson chain is now: ga4x identified the problem, ev3k proved the fix. Mark this pattern as validated.

2. **Test-writing without test-running is a known-unknown.** Writing 79 tests that cannot execute locally means the pipeline is shipping confidence based on code review alone. Tests that have never been green are aspirational, not assurance. The first CI run may surface compilation errors, import mismatches, or environment assumptions that code review cannot catch.

3. **Stakeholder-removed checkpoints should be logged, not debated.** The stakeholder has authority to remove gates. The team's job is to note the decision, continue with maximum diligence, and ensure the retrospective records the tradeoff. This is not a defect -- it is a documented risk acceptance.

4. **UI-only features benefit from Architect skip.** When a feature adds no new infrastructure, no new APIs, no cross-module dependencies, and no new data persistence, the Architect stage adds overhead without insight. The skip criteria should be codified: if all stories are UI-component or config-only, Architect is optional.
