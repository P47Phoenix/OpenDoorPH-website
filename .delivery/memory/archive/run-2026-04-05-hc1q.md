**Type:** BUG_FIX
**Goal:** Fix Terraform CI failures so the new workflow passes
**Status:** Complete — CI green after 3 iterations

## What went well
- Team investigation was thorough — found all 3 root causes (quoted type, legacy state, tags syntax)
- Each fix was isolated and incremental (one commit per issue)
- CI feedback loop was fast (~15s per run)

## What could be improved
- The original SSL cert pipeline (fx8s) pinned Terraform 1.5.7 assuming it was compatible with v0.11 HCL. It wasn't fully — `type = "string"` and `tags {}` are hard errors. The version choice should have been validated against the actual HCL patterns before delivery.
- The state file investigation could have been done during the Terraform workflow feature pipeline (tw3p) rather than discovering it after deployment.

## Lessons learned
1. **Terraform version pinning requires testing against the actual .tf files**, not just assumptions about compatibility. HCL v0.11 patterns that are "deprecated" in docs may be hard errors in practice.
2. **Legacy state files (format v3, Terraform 0.9.x) cannot be read by Terraform 1.x.** Must be removed or migrated through the 0.12→0.13 upgrade path.
3. **Three HCL v0.11 patterns that break on Terraform 1.5.7:** `type = "string"` (quoted type constraints), `tags {}` (block syntax instead of argument), and legacy provider addresses in state. The `${}` interpolation syntax produces warnings but does NOT error.
4. **Iterative CI debugging works well** — push, wait 60s, read logs, fix, repeat. Each cycle was ~2 minutes.

## Defects: 1
- Root cause: Incomplete HCL compatibility testing during Terraform workflow delivery (run tw3p)
- Category: Insufficient pre-deployment validation
