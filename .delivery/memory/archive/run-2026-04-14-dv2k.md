1. **TF-001 apply still hangs over the next pipeline.** CODE_COMPLETE is the correct status here, but the team should not let the apply window slip. KI-TF-001-APPLY targets 2026-05-29; the apply wants a scheduled low-traffic window, a person at the keyboard, and fresh `terraform state pull` + lockfile backup. Calendar this now, do not wait for the target date to approach.
2. **No actionlint in the sandbox.** SV-001's workflow passed a `yaml.safe_load` syntactic check but was not statically linted. `actionlint` as a pre-commit hook (or a CI step) would raise the floor on workflow edits across the repo. Tracked as KI-TF-001-ACTIONLINT (target 2026-05-15) — do not let this one drift.
3. **SV-001 negative tests (T7 redirect-loop, T9 TLS) are designed but not demonstrated.** The gate is built, but we have not yet proven it actually gates. KI-SV-001-NEG-TEST (target 2026-04-26) requires a deliberate broken deploy on a throwaway branch. Schedule this; it is a one-hour exercise that validates the watchman's eyesight.
4. **GA-001 peer walkthrough is the final quality gate.** The runbook is internally consistent and cross-referenced correctly, but a second human (not Ezra) has to walk §8 end-to-end against the live GA4 property before it is truly DONE. KI-GA-001-WALKTHROUGH (target 2026-04-26) is on the property Admin or a designated reviewer. Assign the person now, not on the target date.
5. **`contact_click` decision still owed.** The runbook flagged it as a recommended key event that the code does not emit. A decision — implement or strike — is owed by 2026-05-15 (KI-GA-001-CONTACT-CLICK). Left unresolved, it becomes runbook rot.

## Lessons learned

1. **Bundled DevOps pipelines are efficient when the items are independent.** File-level isolation is the test: if three stories touch disjoint file sets, one pipeline can carry all three with no DoD contention. When file sets overlap, split the pipeline.
2. **CODE_COMPLETE is a first-class status for infrastructure work.** Treating "authored and reviewed by inspection" as a terminal state for the dev stage — and making apply a separately-tracked known issue with a target date — is the right shape. DONE requires production.
3. **Retrospective action items are the backlog for the next pipeline.** bf7w and ga4x both produced concrete action items with owners and targets. dv2k scooped all three into its scope. The retro→backlog→next-pipeline loop is working; keep feeding it.
4. **Zero self-corrections across four stages is a signal of good prior art.** When Idea, Plan, Development, and UAT all pass first-round, either the work was simpler than expected or the inputs were exceptionally well-scoped. Here it was the latter — the prior retros did the scoping work.
5. **State drift must be read, not assumed.** TF-001's `acl: null` / `policy: null` finding only surfaced because the developer actually read `terraform.tfstate`. Future Terraform refactors should include a state audit as a dev-stage checklist item, not an optional read.
